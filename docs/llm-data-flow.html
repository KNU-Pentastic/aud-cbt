공통 봉투 — 모든 LLM 호출이 같은 형태로 게이트웨이를 지난다

아래 모든 LLM 노드는 자신의 입력을 LLMInvokeRequest로 감싸 llm_gateway에 보내고, LLMInvokeResponse로 받습니다. content 안에 각 에이전트의 실제 출력(JSON 문자열 또는 자연어)이 담깁니다.

INLLMInvokeRequest

{
  "model": "claude-haiku-4-5 | sonnet-4-6 | opus-4-7",
  "system": "시스템 프롬프트 | null",
  "messages": [{ "role": "user | assistant", "content": "..." }],
  "max_tokens": 200,
  "temperature": 0.0,            // null 가능
  "stream": false,
  "patient_id": "p_xxx | system",
  "purpose": "safety_classification | module_classification | ...",
  "caller_component": "safety_classifier | module_classifier | ..."
}

OUTLLMInvokeResponse

{
  "content": "에이전트별 출력(아래 표의 JSON) 또는 자연어",
  "usage": { "input_tokens": 0, "output_tokens": 0 },
  "stop_reason": "end_turn",
  "invocation_id": "inv_xxxxxxxxxxxx"
}

＊ purpose·caller_component는 라우팅·과금·관측용 라벨입니다. 게이트웨이는 이 봉투만 알고, 각 에이전트의 의미 구조(아래)는 호출부가 해석합니다.

1

환자 메시지 한 줄이 거치는 데이터 흐름

위에서 아래로 흐릅니다. 각 노드의 왼쪽이 입력, 오른쪽이 출력입니다. ①에서 grade A면 즉시 멈춥니다.

읽는 법 — IN은 그 에이전트가 받는 데이터, OUT은 다음 단계로 넘기는 데이터입니다. 한 노드의 OUT 일부가 다음 노드의 IN으로 이어집니다.
