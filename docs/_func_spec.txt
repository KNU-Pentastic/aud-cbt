1. 문서의 목적과 읽는 방법
1.1 이 문서는 무엇인가
이 문서는 입원 치료를 마친 알코올 사용 장애(Alcohol Use Disorder, AUD — 흔히 '알코올 의존증'으로 알려진 정신건강 진단명) 환자가 퇴원 직후부터 3개월간 사용할 LLM(Large Language Model, 대규모 언어 모델 — ChatGPT 같은 대화형 인공지능) 기반 인지행동치료(CBT, Cognitive Behavioral Therapy — 환자의 생각·감정·행동의 연결을 다루는 검증된 심리치료 기법) 앱의 MVP(Minimum Viable Product, 최소 기능 제품) 기능을 정의합니다.
문서는 다음 두 가지 목적을 동시에 갖습니다:
개발팀이 무엇을 만들어야 하는지 합의하는 기준점
개발팀 안의 비전공자(임상 지식 없는 개발자, 의학 배경 없는 기획·디자인 인력)도 같은 그림을 그릴 수 있도록 임상 용어를 풀어 설명
1.2 본 버전(v3.0)의 핵심 범위

본 MVP는 대회 출품용 시연 데모입니다

v3.0은 대회 데드라인(6월 27일) 안에 시연 가능한 데모 완성을 목표로 v2.0에서 추가 축소되었습니다.
v2.0 대비 축소 사항: 의료진 포털을 환자 입력·조회 최소 기능으로 단순화, JWT Refresh Token 제거, audit log 삭제, 갈망 일지 자동 기록 Post-MVP로 이동, 안드로이드 단일 플랫폼.
의료진 알림 시스템(SMS·푸시·24/7 SLA·백업 의료진)은 v2.0에서 이미 빠졌습니다. 위기 신호 발생 시 환자에게 119/1393 안내가 표시되며, 의료진에게 자동 전달되지 않습니다.
v3.0에서 추가로 단순화된 정책: 안전 분류기 등급 A 발생 후 LLM 잠금 해제는 의료진의 명시적 액션 대신 환자 앱이 시간 기반(24시간 경과) 또는 환자의 명시적 확인으로 처리.
따라서 본 MVP의 사용 범위는 다음에 한정됩니다: 대회 시연, 사내 기술 검증. 실제 환자 서비스 전에는 의료진 알림·잠금 해제 흐름·세부 안전 정책을 다시 구축해야 합니다.

2.3 핵심 가치

이 제품이 해결하는 문제

퇴원 후 첫 3개월은 알코올 의존증 환자가 가장 재발하기 쉬운 시기입니다 (90일 이내 재발률 40~60%, NIAAA 가이드라인 기준). 그러나 외래 진료는 보통 2~4주에 한 번이라 진료 사이의 긴 공백이 발생합니다. 본 앱은 이 공백 동안 환자에게 인지행동치료(CBT)를 매주 제공함으로써 단주 유지율 향상에 기여하는 것을 목표로 합니다.

3. 핵심 임상 개념
이 장은 본 문서의 나머지 부분을 이해하기 위해 필요한 임상 개념을 정리합니다. 이미 알고 있는 독자는 4장으로 건너뛰어도 됩니다.
3.1 알코올 사용 장애와 입원 후 환자의 특성
3.1.1 알코올 사용 장애란
알코올 사용 장애(AUD)는 음주 행동이 본인의 건강·관계·일상에 해를 끼치는데도 음주를 줄이거나 멈추지 못하는 상태를 가리키는 정신건강 진단입니다. 흔히 '알코올 의존증'으로 알려져 있으며, 미국정신의학회의 진단 매뉴얼 DSM-5에서 11가지 증상 중 몇 개에 해당하는지에 따라 경도·중등도·중증으로 분류합니다. 본 앱은 입원 치료가 필요할 정도였던 환자, 즉 중등도~중증을 다룹니다.
3.1.2 입원 후 환자의 특수성
외래 환자가 아니라 '입원 후 환자'를 다룬다는 점이 본 앱의 설계 전반에 영향을 줍니다. 입원 후 환자는 다음과 같은 특성을 갖습니다:
이미 진단이 확정되어 있고 중증도가 명확합니다 (외래 환자처럼 진단부터 시작할 필요가 없음).
입원 중에 급성 금단 증상(술을 끊었을 때 첫 며칠~1주에 나타나는 손떨림·발작·섬망 등)은 의료 환경에서 이미 처리되었습니다. 앱이 마주칠 가능성이 낮습니다.
대신 PAWS(Post-Acute Withdrawal Syndrome, 지연성 금단 증후군 — 수면 장애·집중력 저하·불안·갈망이 수주~수개월 지속되는 상태)가 핵심 관심 대상이 됩니다.
퇴원 후 첫 3개월의 재발 위험이 가장 높습니다 (90일 이내 재발률 40~60%).
자살 위험이 일반 인구의 약 10배 수준입니다. 본 앱은 이 위험을 가장 우선순위 높게 다룹니다.
동반 정신질환(우울·불안)이 매우 흔합니다.
입원 중 알코올 의존 치료 약물(날트렉손·아캄프로세이트·디설피람 등 — 음주 욕구를 줄이거나 음주 시 불쾌감을 유발하는 약)이 처방된 경우가 많고, 복약을 끊으면 재발 위험이 높아집니다.
3.2 인지행동치료(CBT)의 작동 원리
CBT는 환자의 '생각·감정·행동의 연결'을 다루는 검증된 심리치료 기법입니다. 핵심 가정은 다음 한 문장으로 요약됩니다:

CBT의 핵심 가정

같은 사건도 사람마다 다르게 해석하기 때문에, 사건 자체가 아니라 '사건에 대한 해석'을 바꾸면 감정과 행동이 바뀐다.

3.2.1 ABC 모델
CBT가 사용하는 가장 기본 도구입니다. ABC는 다음 세 가지를 뜻합니다:
A (Antecedent, 선행 사건) — 어떤 상황·시간·장소·사람이 있었는가
B (Belief, 해석·생각) — 그 상황을 어떻게 받아들였는가
C (Consequence, 결과 — 감정과 행동) — 어떤 감정이 들었고 어떤 행동을 했는가
예: 회식 자리(A)에서 '한 잔 정도는 괜찮을 거야'(B)라고 생각해 음주(C)했다. CBT는 B 단계의 생각을 바꾸도록 훈련합니다.

무엇을 하는 기능인가  환자의 모든 입력을 받아 5종 위험 신호로 분류하고, 응급도에 따라 등급 A(자살·급성중독)와 등급 B(재발·복약중단·PAWS)로 분리해 다른 처리 경로로 보냄.
왜 필요한가  임상 안전성의 1차 방어선. 응급 상황은 LLM이 아닌 외부 자원으로 명확히 넘기고, 비응급 상황은 LLM이 적절한 분기로 대응.
어떻게 동작하는가
입력 1: 환자 발화 또는 데이터
입력 2: 환자 안전 프로파일 (D0 입력, 자살 사고 이력에 따라 임계값 다름)
처리: 룰 기반 키워드 매칭 + LLM 분류 hybrid (둘 다 동시 실행)
출력: { event_type, severity_grade: 'A'|'B', confidence, raw_matches } JSON
등급 A (자살·급성중독): SafetyEvent 기록 + LLM 응답 중단 + lock 상태 노출 → 환자 앱이 클라이언트 측에서 LLM 대화 진입 차단 → P4 응급 안내 화면으로 강제 이동 (잠금 해제는 환자 앱이 시간 기반 24시간 또는 사용자 명시적 확인으로 처리)
등급 B (재발): RESU 분기 컨텍스트로 전환
등급 B (복약 중단): SOMA 분기 컨텍스트로 전환
등급 B (PAWS): 일반 대화 안에서 인식되며 자료 안내 (별도 분기 없음)
위험 미발견 시: 일반 대화 흐름 계속
주의·한계  Recall(놓치지 않음) &gt; Precision(잘못 잡지 않음). 즉 false negative(위험을 놓침)가 false positive(과대 경보)보다 임상적으로 훨씬 큰 손해. 임계값을 보수적으로 설정. 등급 A의 false positive(잘못 응급 안내)는 환자 경험에 부정적이지만, false negative보다 안전.

6.2 LLM 오케스트레이션 (구조도 레이어 4)

v2.0 LLM 컴포넌트 구성

메인 컨텍스트 3종 (Opus 4.7): 세션 대화 / 갈망 대화 / RESU·SOMA 분기 대화
보조 컴포넌트 3종 (Sonnet 4.6 또는 Haiku 4.5): 단계 판단기 / 세션 요약기 / 안전 분류기 LLM
v1.0에 있던 CRIS(위기 개입), MISS(결석 대응), 콘텐츠 충실도 검증 LLM은 v2.0에서 모두 제거됨.

6.2.1 메인 컨텍스트 3종 운영

분류
MVP 등급
임상 근거 강도

시스템 기능
MVP-핵심
강한 근거

무엇을 하는 기능인가  상황에 따라 3종 메인 컨텍스트(세션·갈망·분기) 중 하나를 활성화하여 LLM 응답 생성.
왜 필요한가  각 컨텍스트는 시스템 프롬프트, 메모리 범위가 모두 다름. 같은 컨텍스트로 모든 상황을 처리하면 임상 일관성 불가.
어떻게 동작하는가
세션 컨텍스트: Week 1~12 중 하나의 프롬프트 + 환자 컨텍스트 + 직전 세션 요약
갈망 컨텍스트: 갈망 대처 프롬프트 + 최근 7일 갈망 로그 + 학습한 대처 기술
분기 컨텍스트(RESU): 재발 대응 프롬프트 + 음주 보고 직전 대화 + 단주 일수 (AVE 차단)
분기 컨텍스트(SOMA): 복약 지원 프롬프트 + 복약 로그 + 환자 처방 정보
컨텍스트 전환은 LLM 단독 결정이 아니라 안전 분류기 결과 + 환자 명시적 선택의 조합
모든 메인 컨텍스트는 Opus 4.7 사용
주의·한계  컨텍스트 분리가 사용자에게 노출되면 안 됨 (자연스러운 전환이 UX 핵심). 동시에 시스템 내부에서는 분리가 엄격해야 함.

6.2.2 Week별 세션 시스템 프롬프트 관리

분류
MVP 등급
임상 근거 강도

시스템 기능
MVP-핵심
강한 근거

무엇을 하는 기능인가  12개 Week의 세션 프롬프트를 버전 관리하며, 각 환자에게 정확한 프롬프트가 전달되도록 함.
왜 필요한가  12 Week × 5 단계 = 60개 단계의 임상 일관성을 보장하려면 프롬프트가 정확히 관리되어야 함.
어떻게 동작하는가
프롬프트 파일은 git 또는 별도 저장소에 버전 관리

프롬프트 변경 시 변경 이력 기록
주의·한계  프롬프트 변경이 임상 효과를 바꾸므로, 운영 중 임의 변경 금지. 변경 후 임상가 시뮬레이션 재검증 필요.

6.2.3 단계 판단기 (Stage Tracker)

분류
MVP 등급
임상 근거 강도

시스템 기능
MVP-핵심
약한 근거/이론적

무엇을 하는 기능인가  CBT 세션 진행 중 환자가 5단계 중 어디에 있는지, 다음 단계로 전환할 준비가 되었는지 판단.
왜 필요한가  세션이 5단계 흐름을 따르도록 강제. 환자가 단계 3(핵심 콘텐츠 전달)을 건너뛰지 않게 함.
어떻게 동작하는가
별도 LLM(Sonnet 4.6) 호출, 매 환자 발화 후 실행
입력: 현재 단계, 현재 세션 대화 로그, 단계별 완료 조건
출력: { current_step: 1-5, ready_to_advance: bool, step_drift_risk: 'low'|'med'|'high' } JSON
ready_to_advance=true이고 단계별 핵심 메시지가 전달되었으면 다음 단계로 전환
환자가 다른 주제로 빠지면 step_drift_risk 상승, 일정 임계값 초과 시 LLM이 부드럽게 복귀 유도
주의·한계  단계 판단기가 또 다른 LLM이므로 환각 가능. 단계 전환 잘못이 큰 임상 손해를 주지 않으므로 우선 단순 구현 후 운영하며 정확도 측정.

6.2.5 안전 분류기 LLM (보조 컴포넌트)

분류

임상 근거 강도

시스템 기능
MVP-핵심
강한 근거

무엇을 하는 기능인가  안전 분류기(레이어 3)가 룰 기반 키워드 매칭만으로 잡지 못하는 미묘한 위험 신호를 LLM이 보완 분류.
왜 필요한가  키워드만으로는 '사라지고 싶다' 같은 명시적 표현은 잡지만, 우회적·은유적 표현은 놓침. LLM 분류로 보완.
어떻게 동작하는가

출력: { event_type, severity_grade, confidence } JSON

주의·한계  Haiku는 빠르지만 정확도가 Opus·Sonnet보다 낮음. confidence 임계값을 보수적으로 설정 (낮은 confidence도 등급 발동).

6.3.1 출력 필터 2종

분류

강한 근거

무엇을 하는 기능인가  LLM이 생성한 응답을 환자에게 보내기 전 2가지 위반 검사. v1.0의 5종에서 v2.0의 2종으로 축소.
왜 필요한가  LLM이 임상적으로 부적절한 응답을 생성할 수 있는 마지막 방어선. 가장 중요한 두 가지(의학 어휘·AVE)만 유지하여 구현 부담 절감.
어떻게 동작하는가
필터 1: 의학 어휘 차단 — 진단·처방·복용량 조절 표현 차단
필터 2: AVE 위반 표현 차단 — '한 잔은 괜찮' 등 절주 권장·음주 합리화 차단
필터는 룰 기반 + Haiku 4.5 LLM 분류 hybrid

재생성도 위반 시: 안전 안내 fallback 메시지로 대체
v1.0의 Sycophancy·세션 표류·위기 부적절 필터는 v2.0에서 제외
주의·한계  축소된 3종 필터(Sycophancy 등)의 부재로 미묘한 LLM 부적절 응답이 환자에게 도달할 가능성 ↑. 단, 임상 시뮬레이션 검증에서 발견 시 프롬프트 개선으로 대응 가능. 운영 데이터 축적 후 필터 추가 검토.

8.4 보안·프라이버시 요구사항
규제는 본 단계에서 고려하지 않지만, 임상 데이터의 민감성 때문에 다음은 v2.0에서도 필수:
환자 데이터 전송 시 TLS 1.2 이상
저장 시 데이터베이스 암호화 (at-rest encryption)
의료진 로그인 시 2단계 인증
환자 식별 정보와 임상 데이터 분리 저장
LLM API 요청 시 환자 식별 정보 제외 (해시된 ID만 전송)
대화 원본은 환자 본인만 접근(GET /me/conversations/.../messages). v3.0에서 의료진의 대화 원본 조회 엔드포인트는 audit log 삭제와 함께 Post-MVP로 이동됨. 실 환자 서비스 단계에서는 audit log를 복원하여 ‘접근자·시각·범위’를 모두 기록해야 함(9.1 Post-MVP 표 참조).
